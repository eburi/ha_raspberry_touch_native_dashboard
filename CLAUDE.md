## Quick Reference

```bash
# Zig is at .local/bin/zig (symlink to .local/zig/zig-aarch64-linux-0.14.1/zig)
export PATH="$PWD/.local/bin:$PATH"

# Compile everything (WASM + server)
zig build

# Run tests
zig build test

# Build individual targets
zig build wasm              # WASM module only
zig build server            # Native server only

# Optimized builds (as CI runs)
zig build test -Doptimize=ReleaseSafe
zig build wasm -Doptimize=ReleaseSmall
zig build server -Doptimize=ReleaseSafe

# Format code (CI enforces this)
zig fmt src/ tests/ build.zig
zig fmt --check src/ tests/ build.zig   # check only, no changes
```

## Project Overview

A Home Assistant **App** (not "add-on" — always use "app" terminology) that renders a
touch-screen dashboard using the LVGL graphics library on a Raspberry Pi with an
RPi Touch Display 2 (1280x720). Runs on Home Assistant OS (HAOS) inside a Docker
container with direct access to `/dev/fb0` (framebuffer) and `/dev/input/event0`
(touch input).

### Key Architectural Constraint

HAOS runs busybox + Docker with no graphical environment. The app runs in a
container and accesses display hardware directly via device passthrough. The WASM
web version exists so development can proceed without hardware access — it is also
the long-term browser/ingress interface.

### Two Rendering Paths

1. **WASM (browser)**: LVGL compiled to WebAssembly, rendered on HTML5 Canvas via
   JavaScript glue (`web/main.js`). Each browser client runs its own LVGL instance.
2. **Native (RPi)**: LVGL rendering directly to `/dev/fb0` with touch input from
   `/dev/input/event0`. Integrated into the server binary — the same process serves
   the web UI and drives the physical display when hardware is detected.
   - On Mac/dev: probe finds no `/dev/fb0`, runs web-only mode
   - On RPi: renders to framebuffer + serves web interface simultaneously

## Language & Build System

- **Zig 0.14.1** (required exact minor version — 0.15 breaks zap/facil.io)
- Zig's native build system (`build.zig` + `build.zig.zon`)
- Dependencies: LVGL v9.2.2, Zap v0.10.6 (HTTP server framework)
- Tests use Zig's built-in `std.testing` framework

## Important Conventions

### Version Bumps
Every pushed change **must** include a version bump in
`ha_raspberry_touch_native_dashboard/config.yaml`. Use [Semantic Versioning](https://semver.org/):
- **MAJOR** (`X.0.0`): Breaking changes (incompatible config, removed features)
- **MINOR** (`0.X.0`): New features, new user-facing functionality
- **PATCH** (`0.0.X`): Bug fixes, internal refactors, dependency updates, doc changes

Include the version bump in the same commit as the code change — do not create
a separate "bump version" commit.

### Terminology
- Always say "Home Assistant **App**", never "add-on" or "addon"
- CLI commands: `ha apps install`, `ha apps start`, `ha apps logs` (not `ha addons`)

### std_options (Zig 0.14.x)
`std_options` must be declared as a **value** of type `std.Options`, not a type:
```zig
// CORRECT (Zig 0.14.x):
pub const std_options: std.Options = .{ .log_level = .debug };

// WRONG (old style, causes "expected type 'std.Options', found 'type'"):
pub const std_options = struct { pub const log_level = .debug; };
```

### Catch expressions
When using a catch block that should evaluate to a value, don't wrap it in braces
that create a void block. Either use a plain expression or a labeled block with break:
```zig
// CORRECT:
const x = tryFoo() catch tryBar() catch |err| { return null; };

// WRONG (void block, missing semicolon error):
const x = tryFoo() catch { tryBar() catch |err| { return null; } };
```

### Pixel Format
LVGL renders XRGB8888 (little-endian: B,G,R,X in memory). HTML Canvas expects RGBA.
The WASM display flush callback in `display.zig` swaps R<->B and sets A=0xFF.
The native framebuffer path uses XRGB8888 directly — no pixel conversion needed.

### Adding Tests
Add new test files under `tests/native/` or `tests/server/`, then add a comptime
import in `tests/test_main.zig`. If the test needs a module import, add it in
`build.zig` under the test target section.

### Adding Icons
1. Add icon name to `icons_list.txt` (one per line, Tabler icon name or `assets/path.svg`)
2. Run: `.venv/bin/python tools/install_icons.py icons_list.txt`
3. This regenerates `src/generated_icons/tabler_icons.{c,h}`
4. Requires Python venv with `cairosvg` and `pillow`

## Deployment

### Local Deploy to HA Instance
```bash
tools/deploy.sh [user@host]        # default: root@192.168.46.222
```
Assembles the app directory, copies via SCP, then rebuilds and starts the app
on the HA instance.

### Dev Server
```bash
./dev.sh                                    # web-only mode
./dev.sh --ha-host root@192.168.46.222      # with HA tunnel for live data
```
Builds both targets, starts the server on port 8765, watches for file changes,
and auto-rebuilds. When `--ha-host` is given, it steals the SUPERVISOR_TOKEN
from the running app container and creates an SSH tunnel to the HA supervisor API.

## Common Pitfalls

- Zig 0.15 is **incompatible** — it changes the build API and breaks zap/facil.io
- The WASM target is `wasm32-freestanding` (no libc) — `src/wasm/libc.zig` provides stubs
- `web/dashboard.wasm` is gitignored — always rebuild with `zig build wasm`
- `src/generated_icons/` is checked in (generated C code, not gitignored)
- The server reads config from env vars: `PORT`, `WEB_ROOT`, `LOG_LEVEL`,
  `SUPERVISOR_TOKEN`, `HA_URL`, `SIGNALK_URL`
- Shared modules (`lv`, `input`, `dashboard`) are created separately per target in
  `build.zig` — one set for WASM, another for native/server. Each needs LVGL include paths.
- `dashboard.zig` uses `@import("lv")` (module name), not `@import("lv.zig")` (relative path),
  to avoid "file exists in multiple modules" errors in Zig 0.14
- `dashboard.zig` uses `PlatformCallbacks` struct instead of `extern fn` — each platform
  sets its callbacks via `setPlatformCallbacks()` (WASM wraps JS externs, native calls HA)
- LVGL is **not thread-safe** — the native path runs LVGL in a dedicated background thread
- `lv_conf.h` works for both targets; `LV_STDLIB_BUILTIN` provides LVGL's own malloc on both
- `std.mem.page_size` does not exist in Zig 0.14 — use `std.heap.page_size_min` instead
- Do **not** set `ingress_entry: "/"` in `config.yaml` — `/` is already the default,
  and setting it explicitly used to cause double-slash (`//`) paths from HA's ingress
  proxy. The HA docs are misleading on this. Simply omit `ingress_entry` entirely.
